## 2. 📋 贡献方式

### 2.1. 报告问题

如果您发现了文档中的错误、不准确的内容或需要改进的地方，请：

1. 检查是否已有相关issue
2. 创建新的issue，详细描述问题
3. 提供具体的文件路径和问题位置

### 2.2. 提交改进

如果您想改进文档内容：

1. Fork本项目
2. 创建新的分支
3. 进行修改
4. 提交Pull Request

### 2.3. 添加内容

如果您想添加新的内容：

1. 确保内容符合项目规范
2. 参考现有文档的格式
3. 添加适当的交叉引用
4. 更新相关索引文件

---

## 3. 📝 文档规范

### 3.1. 格式要求

所有文档应遵循 [文档格式规范](./00-项目导航/03-规范模板/文档格式规范.md)：

1. **目录结构**：必须包含目录（TOC）
2. **编号规范**：主题和子主题使用统一编号
3. **格式一致**：标题、代码块、列表等格式统一
4. **交叉引用**：添加适当的内部链接

### 3.2. 内容要求

1. **准确性**：内容必须准确，有据可查
2. **完整性**：覆盖主题的各个方面
3. **实用性**：提供实用的示例和最佳实践
4. **可读性**：语言清晰，结构合理

### 4.1. 工具要求

## 5. 📚 内容分类

### 5.1. 核心文档

核心文档位于 `01-核心架构/` 至 `09-最新特性/` 目录：

- 必须遵循严格的格式规范
- 需要包含完整的目录和编号
- 需要添加交叉引用

### 5.2. 代码示例

代码示例位于 `examples/` 目录：

- 必须可运行
- 需要包含详细注释
- 需要包含错误处理

### 5.3. 实战案例

实战案例位于 `cases/` 目录：

- 基于真实应用场景
- 包含技术架构和性能数据
- 提供最佳实践总结

### 5.4. 工具脚本

工具脚本位于 `tools/` 目录：

- 必须可执行
- 需要包含使用说明
- 需要包含错误处理

## 9. 📝 提交规范

### 9.1. Commit Message

使用清晰的commit message：

```text
类型: 简短描述

详细描述（可选）

- 修改点1
- 修改点2
```

**类型**：

- `fix`: 修复错误
- `feat`: 新功能
- `docs`: 文档更新
- `refactor`: 重构
- `style`: 格式调整

### 9.2. Pull Request

PR应包含：

1. 清晰的标题和描述
2. 修改的文件列表
3. 测试结果（如适用）
4. 相关issue链接（如适用）

2. 遵循命名规范：`XX.XX-主题名称.md`

4. 更新INDEX.md和相关README

### 11.2. Q: 如何添加代码示例？
