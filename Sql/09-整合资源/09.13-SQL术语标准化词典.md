## 事务与并发术语

### ACID术语

| 英文 | 中文 | 缩写 | 错误翻译 | 解释 |
|------|-----|------|---------|------|
| Atomicity | 原子性 | A | 原子化 | 全有或全无 |
| Consistency | 一致性 | C | 连续性 | 满足所有约束 |
| Isolation | 隔离性 | I | 孤立性 | 并发事务不互相影响 |
| Durability | 持久性 | D | 耐久性 | 提交后永久保存 |
| Transaction | 事务 | Txn | 交易 | 一组原子操作 |
| Commit | 提交 | - | 确认 | 持久化事务 |
| Rollback | 回滚 | - | 撤销 | 撤销事务 |
| Abort | 中止 | - | 异常终止 | 事务失败 |

### 并发控制术语

| 英文 | 中文 | 缩写 | 避免 | 说明 |
|------|-----|------|------|------|
| Concurrency | 并发 | - | 并行 | 逻辑上同时 |
| Parallelism | 并行 | - | 并发 | 物理上同时 |
| Lock | 锁 | - | 锁定 | 同步原语 |
| Deadlock | 死锁 | - | 僵局 | 循环等待 |
| Livelock | 活锁 | - | 活动锁 | 一直重试 |
| Starvation | 饿死 | - | 饥饿 | 永远等待 |
| MVCC | 多版本并发控制 | - | 多版本 | Multi-Version CC |
| 2PL | 两阶段锁 | - | 二段锁 | Two-Phase Locking |
| Snapshot Isolation | 快照隔离 | SI | - | 基于快照的隔离 |
| Serializable | 可串行化 | - | 串行化 | 等价于串行执行 |

**辨析**:

```text
并发 vs 并行
────────────────────────────────────
并发（Concurrency）:
• 逻辑概念
• 多个任务交替执行
• 单核CPU可实现
• 示例：时间片轮转

并行（Parallelism）:
• 物理概念
• 多个任务同时执行
• 需要多核CPU
• 示例：多线程排序

数据库场景:
• "并发事务"：逻辑上同时执行（可能串行）
• "并行查询"：物理上多线程同时执行

死锁 vs 活锁
────────────────────────────────────
死锁（Deadlock）:
• 循环等待资源
• 进程停滞
• 需要检测和解除

活锁（Livelock）:
• 一直重试但无进展
• 进程活跃（CPU消耗）
• 需要随机退避

饿死（Starvation）:
• 永远得不到资源
• 公平性问题
• 需要优先级调度
```

**辨析**:

```text
查询计划 vs 执行计划
────────────────────────────────────
查询计划（Query Plan）:
• 优化器生成
• 逻辑计划
• 可能有多个候选

执行计划（Execution Plan）:
• 优化器选择的最优计划
• 物理计划
• 实际执行的路径

通常混用，但严格区分:
• EXPLAIN输出：执行计划
• 优化器生成：查询计划（多个）

选择性 vs 基数
────────────────────────────────────
选择性（Selectivity）:
• 比例值（0-1）
• 过滤后/过滤前
• 示例：WHERE age > 18，选择性=0.8（80%行满足）

基数（Cardinality）:
• 绝对值
• 行数
• 示例：users表基数=100万行

关系：基数 = 选择性 × 总行数
```

## 易混淆术语辨析

### Schema相关

```text
Schema的四种含义
══════════════════════════════════════════════════════════════════════════════

1. 关系模式（Relation Schema）
   定义：R(A₁, A₂, ..., Aₙ)
   示例：users(id, name, email)
   语境：理论讨论

2. 数据库模式（Database Schema）
   定义：所有表的结构定义
   示例：整个数据库的DDL
   语境：数据库设计

3. 命名空间（Namespace）
   定义：PostgreSQL的schema概念
   示例：CREATE SCHEMA app;
   语境：PostgreSQL实践

4. 数据库（Database）
   定义：MySQL中的database = PostgreSQL的schema
   示例：CREATE DATABASE mydb; (MySQL)
   语境：MySQL实践

对应关系:
PostgreSQL     | MySQL
---------------|----------------
Database       | (Cluster/Instance)
Schema         | Database
Table          | Table

避免混淆:
• 说"表结构"而非"表schema"
• 说"数据库设计"而非"schema设计"
• 明确上下文（PostgreSQL/MySQL）
```

### Index相关

```text
Index术语辨析
══════════════════════════════════════════════════════════════════════════════

聚簇索引 vs 聚集索引（同义）:
• 英文：Clustered Index
• 定义：数据按索引顺序物理存储
• 特点：每表只能有一个
• 示例：InnoDB主键

二级索引 vs 辅助索引 vs 非聚簇索引（同义）:
• 英文：Secondary Index / Non-Clustered Index
• 定义：独立于数据的索引
• 特点：存储 (索引键, 行指针)
• 示例：InnoDB非主键索引

覆盖索引（两种含义）:
1. 索引包含查询所需所有列（Covering Index）
   • 无需回表
   • 性能提升

2. PostgreSQL的INCLUDE子句
   • CREATE INDEX idx ON t(a) INCLUDE (b, c);
   • 精确控制包含列

统一使用:
• "覆盖索引"指概念（包含查询列）
• "INCLUDE索引"指PostgreSQL语法
```
